Connection to HAAK / Beowolff infrastructure

This document maps existing technical work in the HAAK / Beowolff codebase to Plataforma's stack. Nothing is committed. The connection is adjacent technical-infrastructure overlap; if Plataforma wants to use any of this, it would be a deliberate adoption decision by the founders, not an inheritance.

What transfers cleanly

The auto-populate substrate

The MoMA-Wikidata resolver is exactly the kind of tool the founders' concept names ("the platform automatically searches for your existing identifiers and connects them"). It generalizes to any museum that has a Wikidata external-ID property assigned. A Plataforma adoption would:

  1. Run the resolver across all major museum source-properties (Met, NGA, Rijksmuseum, AIC, Cleveland, V&A, Smithsonian, etc.)
  2. Build per-artist join tables: artist → list of museum-attributed works → Wikidata QID → Commons image URL where available
  3. Surface "we found N of your works in collections" to artists at sign-up

This is days of work, not weeks, given the resolver pattern is established.

The web adapter pattern

For artists whose work isn't in museum collections (most contemporary artists), the gallery-website + artist-personal-site path covers what museum APIs miss. The Artlogic + Squarespace adapters reach a meaningful fraction of contemporary art's visual presence online.

Plataforma adoption would: per-artist, attempt to crawl the artist's representing gallery (if Artlogic-hosted) and personal site (if Squarespace-hosted). Yield a candidate work list for the artist's confirmation — same UX the founders' doc names ("we found 47 of your works"), just with a different source.

The identity layer (ORCID-for-artists)

The ULAN / ISNI / Wikidata QID / MusicBrainz reconciliation pattern is what the technical extension doc spells out. Already implemented for Wikidata (in the resolver); the others are the same code shape with a different external-ID property. ~2-3 days per identifier.

What doesn't transfer

The bot UX layer

The Beowolff Telegram bot is not the Plataforma surface. It's a demonstration vehicle for visual similarity, designed for one-image-at-a-time analysis on mobile. Plataforma's surface is a directory — multi-artist, multi-work, browsable, persistent. Different product entirely.

If Plataforma adopts Beowolff's engine, it adopts the FastAPI sidecar (the engine) and re-skins the front-end (the directory + artist pages) from scratch. The bot retires or stays as a side product.

The pitch context

The Beowolff pitch to Andrew Wolff / Jeffrey Yin (per home/collabs/beowolff/pitch/job.md) frames Beowolff as an integration into Artsy/Artnet's existing inventory. This framing is incompatible with Plataforma's cooperative-alternative framing. A team building Plataforma can't simultaneously pitch its core technology as the recommendation layer for the commercial discovery incumbent — at least not without careful sequencing.

The honest answer is probably: the same technical core can serve both, but the story is different per audience. Beowolff Capital sees "embedding-grade recommendation for the merged Artsy/Artnet catalog." Plataforma members see "embedding-grade recommendation built on infrastructure artists own." Both are true; they're separate products with shared substrate.

The closed-license / negotiated-access problem

Beowolff's "internal-demo posture" deals with the closed-license problem by ingesting closed-license sources (MoMA-direct, Tate-direct, Whitney-direct, etc.) under research-use framing. Plataforma can't do this because Plataforma is built around artist data sovereignty — ingesting closed-license artist work into a member-facing directory recreates the extraction it claims to reject.

Plataforma's path on closed-license is different: the artist who joins explicitly grants the license. The platform doesn't scrape; it asks. The MoMA-direct / Tate-direct ingest path is for Beowolff's separate use case, not for Plataforma.

What's speculative

Whether the visualiser pattern fits

The 690k-work pan-and-zoom map is gorgeous as a Beowolff demo but might be the wrong UX for Plataforma. Plataforma's organizing entity is the artist, not the work. The right primary view is probably an artist page (with their full catalog), with relationships to other artists rendered as a network — closer to Discogs's artist page than to Pinterest's infinite scroll.

The visualiser could appear as a "show me the field of art" affordance — a way to see the embedding space — but it's secondary to the artist directory.

Whether the Wolff pitch is leverage or constraint

If Beowolff Capital (Andrew Wolff + Jeffrey Yin's merged Artsy/Artnet entity) becomes a Plataforma backer or partner, the technical work has compounding value. If the relationship stays purely commercial (Beowolff as a discrete recommendation product, Plataforma as a separate cooperative), the technical work is parallel but politically separate.

The right answer probably depends on whether Wolff/Yin would back a cooperative directory project that explicitly positions itself as upstream of the commercial discovery layer. That's a conversation, not a technical question.

Whether "Plataforma" is the right name

The May extension flagged this in Appendix C. The March doc uses "Plataforma" as a working name; the project will need a final name that "an artist would be proud to list on their CV." A few candidate axes:

  • Latin / cultural seriousness (e.g. "Plataforma," "Atelier," "Codex")
  • ORCID-style technical naming (acronym: AIC, AVCN, etc.)
  • Aspirational / mission-evoking (e.g. "Common Catalog," "Open Atlas")

The name decision affects URL choice and brand identity downstream. Worth converging early.
